# 그룹 권한 및 회차 기록 설계

## 기준

2026-05-30 기준 그룹 권한은 owner/admin/member 구조로 설계한다.

- owner: 그룹을 만든 사용자.
- admin: owner가 권한을 부여한 사용자.
- member: 그룹에 참여한 일반 구성원.

이 구조는 빠르게 구현할 수 있으면서도, 그룹 생성자가 다른 구성원에게 운영 권한을 줄 수 있어 확장성이 있다.

## 역할별 권한

### owner

- 그룹 정보 수정.
- 그룹원 초대.
- 그룹원 관리.
- 그룹원 관리 화면에서 admin 권한 부여 및 회수.
- 그룹 삭제.
- 회차 추가.
- 회차별 기록 관리.
- 부적절하거나 잘못 올라온 게시글 삭제.

### admin

- 그룹원 초대.
- 그룹원 관리.
- 회차 추가.
- 회차별 기록 관리.
- 부적절하거나 잘못 올라온 게시글 삭제.

admin은 그룹 운영을 도울 수 있지만, 다른 사람에게 admin 권한을 부여하거나 그룹 자체를 삭제할 수는 없다. 이 두 기능은 owner만 사용할 수 있다.

### member

- 그룹 피드 읽기.
- 회차별 기록 목록 보기.
- 열린 회차에 짧은 영상 또는 사진 업로드.
- 자기 글귀 삽입, 수정, 삭제.
- 자기 게시글 수정.

member는 그룹 운영 권한을 갖지 않는다. 그룹원 관리, 권한 부여, 그룹 삭제, 회차 추가 기능은 사용할 수 없다.

## 화면에 필요한 기능

### 그룹 화면

- 그룹원 초대.
- owner/admin 전용 그룹원 관리.
- owner 전용 권한 부여.
- owner 전용 그룹 삭제.
- owner/admin 전용 회차 추가.

회차 추가는 야구 경기 흐름에 맞춰 `1회`, `2회`, `3회`, ..., `n회` 형태로 만든다. owner와 admin만 새 회차를 만들 수 있다.

2026-05-30 기준 구현 상태:

- 하단 탭의 `업로드` 자리를 `그룹`으로 교체했다.
- `GroupsFragment`에서 팀별 그룹 목록을 조회한다.
- 사용자가 새 그룹 이름을 입력해 그룹을 만들 수 있다.
- 그룹 카드의 `입장` 버튼으로 선택한 그룹 상세 영역에 들어간다.
- owner/admin은 그룹원 초대, 그룹원 관리, 회차 추가를 사용할 수 있다.
- owner는 admin 권한 부여/회수와 그룹 삭제를 사용할 수 있다.
- member는 그룹 상세, 구성원 목록, 회차 목록을 읽는 흐름으로 제한된다.
- 회차 추가 후 회차 목록에 `1회`, `2회` 같은 항목을 표시한다.
- 회차 항목의 `회차 상세` 버튼을 누르면 구성원별 기록 화면으로 들어간다.
- 그룹 상세의 회차 목록은 가로 스크롤 카드 형태로 표시한다.
- 각 회차 카드 안에는 구성원 이름과 해당 회차 기록 요약을 함께 보여준다.
- 회차 상세 화면에서 현재 사용자는 자기 기록의 글귀를 작성/수정할 수 있고, 사진 또는 5초 이하 영상을 선택해 저장할 수 있다.
- 기존 기록이 있으면 내 기록 삭제 버튼으로 해당 회차 기록을 삭제할 수 있다.
- owner/admin은 구성원 기록 카드에서 다른 구성원의 기록을 삭제할 수 있다.
- 다음 단계에서는 회차 상세 화면을 RecyclerView 기반 전용 화면으로 분리하고, 카드 안의 미디어 URL을 실제 이미지/영상 미리보기로 보여주는 작업이 필요하다.

### 회차 상세 화면

회차가 추가되면 해당 회차 상세 화면에서 그룹 구성원 내역을 볼 수 있어야 한다.

구성원 수만큼 리스트 항목을 보여주고, 각 항목에는 해당 구성원의 회차 기록을 표시한다.

- 짧은 영상 또는 사진.
- 글귀.
- 글귀 삽입.
- 글귀 수정.
- 글귀 삭제.

사용자는 기본적으로 자기 기록만 작성, 수정, 삭제할 수 있다. owner/admin은 관리 목적상 다른 사용자의 부적절한 기록을 삭제할 수 있다.

## 게시글 삭제 기준

게시글 삭제는 단순히 화면에서 숨기는 것이 아니라, 해당 경기일과 회차에 남긴 그 사용자의 기록을 삭제하는 것으로 처리한다.

예시:

- 2026-05-30 경기의 3회차 기록에 사용자가 사진과 글귀를 올렸다.
- 해당 게시글을 삭제하면 그 사용자의 3회차 사진/영상과 글귀 기록이 함께 사라진다.

그룹 전체 회차나 다른 구성원의 기록은 삭제하지 않는다. 삭제 대상은 해당 사용자의 해당 회차 기록이다.

## 권장 Firestore 구조

```text
groups/{groupId}
- ownerId
- adminIds
- name
- teamId
- createdAt

groups/{groupId}/members/{userId}
- role: "owner" | "admin" | "member"
- joinedAt

groups/{groupId}/innings/{inningId}
- inningNumber
- createdBy
- createdAt

groups/{groupId}/innings/{inningId}/records/{userId}
- authorId
- mediaType: "image" | "video"
- mediaUrl
- text
- createdAt
- updatedAt
```

`records/{userId}`처럼 사용자별 기록 문서를 두면, 한 회차에서 한 구성원이 하나의 기록을 갖는 구조가 된다. 회차 상세 화면에서 구성원 목록과 기록을 매칭하기 쉽고, 삭제 시 해당 사용자의 그 회차 기록만 지우기 쉽다.

## 권장 Firestore Security Rules 범위

초기 구현에서는 앱 화면의 버튼 노출과 클라이언트 검증을 먼저 구현한다. Firebase 연동이 안정화되면 다음 수준의 Security Rules를 적용한다.

- 로그인한 사용자만 그룹 데이터를 읽고 쓸 수 있다.
- 그룹 구성원만 해당 그룹의 회차와 기록을 읽을 수 있다.
- owner/admin만 회차를 추가할 수 있다.
- member는 자기 `records/{userId}` 문서만 만들고 수정할 수 있다.
- owner/admin은 관리 목적으로 기록을 삭제할 수 있다.
- owner만 admin 권한을 부여하거나 회수할 수 있다.
- owner만 그룹을 삭제할 수 있다.

2026-05-30 기준 앱 코드에는 `GroupPermissionPolicy`를 추가해 클라이언트 쪽 권한 판단을 한 곳에 모았다. 실제 Firebase 보안 강제는 이후 Firestore Security Rules 작성 단계에서 추가해야 한다.

## 영상 길이 제한

5초 이하 영상 제한은 빠른 구현을 위해 클라이언트에서 우선 처리한다.

앱에서 사용자가 영상을 선택했을 때 영상 길이를 확인하고, 5초를 초과하면 업로드를 막는다. 이 방식은 구현이 빠르고 사용자에게 바로 피드백을 줄 수 있다.

다만 클라이언트 검증은 앱을 우회한 업로드까지 완전히 막지는 못한다. 추후 실제 서비스로 확장하거나 보안 요구가 커지면 Cloud Functions 등을 이용해 서버에서 영상 길이를 한 번 더 검증하는 방식으로 업데이트할 수 있다.

2026-06-02 기준 앱 코드에는 `VideoUploadPolicy`를 유지하고, 별도 업로드 탭이 아니라 `GroupsFragment`의 회차 기록 미디어 선택에서 5초 초과 영상을 차단하도록 연결했다.

## 2026-06-04 실제 기기 검증 결과

Android 기기 `RFCW60Z3FAB`에서 그룹 생성 흐름을 검증했다.

- `응원 그룹 만들기` 버튼 클릭 후 새 그룹 이름 입력 폼이 열리는 구조로 바꿨다.
- 그룹 생성 문서는 `memberIds` 배열을 포함하고, 목록 조회도 `memberIds` 기반으로 바꿨다.
- `QAGroup` 생성 시도 결과 Firestore 로그에서 `Write failed at groups/...: PERMISSION_DENIED`가 확인됐다.
- 즉, 앱 UI 입력 흐름은 동작하지만 실제 생성은 Firebase Firestore Security Rules 배포가 막고 있다.

이 결과에 따라 [Firebase MVP Rules Update](firebase-mvp-rules-2026-06-04.md)를 추가했다. 기존의 "Firebase 연동이 안정화되면 이후 Security Rules 작성"이라는 주장은 현재 생성 기능을 막는 원인이 되었으므로, MVP 기준에서 rules 배포가 필수 작업으로 승격된다.
