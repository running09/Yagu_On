Yagu_On 아키텍처 비교

현재 Android Java/XML + Firebase MVP 구조를 기준으로, 그룹/회차/권한/미디어 업로드 요구를 얹었을 때의 위험과 권장 방향을 비교한다.

결론부터

현재 구조는 빠른 시연용 MVP로는 괜찮지만, 그룹/회차/권한 기능을 그대로 얹기에는 부족하다. 특히 기존 posts 중심 데이터 구조는 “한 그룹 안에서 owner/admin이 회차를 열고, 구성원별 사진/영상/글귀를 기록한다”는 요구와 맞지 않는다. 화면을 더 만들기 전에 권한 정책과 회차 기록 데이터 구조를 먼저 분리해야 한다.

2026-05-30 구현 반영: GroupPermissionPolicy, VideoUploadPolicy, 그룹/회차 모델, Repository 계약, Firebase 구현체 기반, RepositoryProvider를 추가했다. 기존 화면 전체를 패키지 이동하지는 않았고, 새 기능이 붙을 안정적인 기반부터 분리했다.

현재 구조 요약

UI

Activity / Fragment

  • AuthGateActivity, LoginActivity, SignupActivity, TeamSelectActivity
  • MainActivity가 하단 탭과 팀 선택 칩을 관리
  • HomeFragment, CheerFragment, GroupsFragment, SongsFragment, ProfileFragment
Data

Repository + Firebase

  • AuthRepository, UserRepository, PostRepository 등 인터페이스 존재
  • FirebaseAuthRepository, FirebasePostRepository 등 Firebase 구현체 존재

Model / Util

모델과 유틸

  • UserProfile, Post, Team, CheerMessage

  • Ui 유틸은 코드 기반 View 생성을 담당

현재 구조의 장단점

장점
  • 파일 수가 적고 구조가 단순해서 처음 이해하기 쉽다.
  • Repository 인터페이스를 둔 점은 좋은 출발이다.
  • Firebase Auth, Firestore, Storage까지 MVP 기능 연결이 이미 되어 있다.
  • Java/XML 과제 프로젝트로 설명하기 쉽다.
  • 팀 선택, 로그인, 업로드, 응원방 같은 큰 화면 뼈대가 이미 있다.
단점
  • Activity/Fragment가 Repository 구현체를 직접 생성한다. 테스트와 교체가 어렵다.
  • 화면 코드가 UI 구성, 입력 검증, Firebase 요청, 이동 처리를 동시에 한다.
  • 권한 판단을 둘 곳이 없다. owner/admin/member 조건문이 화면마다 퍼질 가능성이 크다.
  • posts 컬렉션은 팀 피드에는 맞지만 그룹/회차/구성원 기록에는 안 맞는다.
  • LinearLayout에 View를 계속 추가하는 방식은 구성원 리스트와 기록 리스트가 커지면 금방 한계가 온다.
  • 웹 크롤링 데이터가 들어올 계층이 없다. 앱 화면에서 바로 긁기 시작하면 유지보수가 나빠진다.

현재 방식 vs 제안 방식

항목 현재 방식 제안 방식 판단
패키지 구조 루트 패키지에 Activity/Fragment가 몰려 있음 ui, domain, data, model로 분리 지금은 괜찮지만 기능 추가 시 금방 지저분해진다.
권한 처리 아직 명확한 권한 정책 코드 없음 GroupPermissionPolicy에서 owner/admin/member 판단 반드시 분리해야 한다. 안 그러면 버그가 화면마다 생긴다.
게시글 구조 posts 컬렉션에 팀별 게시글 저장 groups/{groupId}/innings/{inningId}/records/{userId} 새 요구에는 제안 구조가 맞다.
목록 화면 LinearLayout에 카드 추가 RecyclerView Adapter 사용 회차 상세와 구성원별 기록은 RecyclerView가 맞다.
영상 제한 아직 명확한 정책 코드 없음 VideoUploadPolicy에서 5초 제한 검사 클라이언트 우선 제한 결정을 코드에서도 한 곳에 모아야 한다.
웹 크롤링 데이터 아직 계층 없음 GameDataRepository + Firestore 캐시 앱에서 직접 크롤링은 최후의 수단이다.

그룹/회차 기능 데이터 방향

groups/{groupId}
- ownerId
- adminIds
- name
- teamId

groups/{groupId}/members/{userId}
- role: "owner" | "admin" | "member"

groups/{groupId}/innings/{inningId}
- inningNumber
- createdBy
- createdAt

groups/{groupId}/innings/{inningId}/records/{userId}
- authorId
- mediaType: "image" | "video"
- mediaUrl
- text
- createdAt
- updatedAt

이 구조는 “회차 추가 이후 구성원 내역 리스트를 보고, 구성원별 짧은 영상/사진/글귀를 올린다”는 화면 요구에 맞다. 기존 posts는 팀 전체 피드용으로 남기고, 그룹 회차 기록은 새 구조로 분리하는 편이 낫다.

우선순위

  1. 그룹/회차/기록 모델부터 만든다. 기존 posts를 억지로 확장하지 않는다.
  2. GroupPermissionPolicy를 만든다. owner/admin/member 판단을 화면에 흩뿌리지 않는다.
  3. 회차 상세 화면은 RecyclerView로 간다. 구성원 수만큼 반복되는 화면은 RecyclerView가 맞다.
  4. VideoUploadPolicy로 5초 제한을 한 곳에 둔다. 빠른 구현은 클라이언트에서 하되 나중에 서버 검증으로 확장 가능하게 한다.
  5. 웹 크롤링 결과는 Firestore 캐시를 거쳐 앱에서 읽는다. 앱 화면에서 직접 크롤링하는 구조는 오래 못 간다.

솔직한 리스크

리스크 1

현재 구조로 계속 붙이면 중복이 늘어난다

권한 확인, 로딩 처리, 오류 메시지, Firebase 경로가 화면마다 복사될 가능성이 높다.

리스크 2

데이터 구조를 늦게 바꾸면 더 아프다

팀 피드용 posts로 그룹 회차 기록을 구현하면 나중에 화면 요구와 충돌한다.

리스크 3

크롤링은 앱 핵심 로직이 되면 안 된다

크롤링 대상 사이트 구조가 바뀌면 앱 기능이 깨진다. 캐시 계층이 필요하다.
