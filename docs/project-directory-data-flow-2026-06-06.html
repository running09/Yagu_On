YAGU_ON / 응원ON 구조 설명

디렉터리별 역할과 데이터 흐름

이 프로젝트는 Android Java/XML 앱을 빠르게 Firebase MVP로 만들기 위해 화면, 도메인 규칙, 데이터 계약, Firebase 구현체를 나눠 둔 구조다. 핵심 데이터는 화면에서 바로 Firebase를 만지지 않고 Repository 계층을 지나가도록 구성되어 있다.

전체 흐름 한 장 요약

User

사용자 입력

로그인, 팀 선택, 하단 탭 이동, 그룹 생성, 회차 기록 저장 같은 행동이 시작점이다.

UI

Activity / Fragment

MainActivity가 공통 프레임을 들고, 각 Fragment가 화면별 입력과 렌더링을 맡는다.

Rule

Domain

폼 검증, 권한 판단, 사용자용 오류 문구, 5초 영상 제한 같은 앱 규칙을 처리한다.

Data

Repository

화면은 인터페이스만 호출한다. 실제 Firebase 접근은 구현체가 맡는다.

Firebase

Auth / Firestore / Storage

인증, 사용자 프로필, 그룹, 회차 기록, 미디어 파일이 원격 DB와 Storage에 저장된다.

디렉터리별로 왜 이렇게 나눴는가

app/

Android 앱 모듈

실제 APK로 빌드되는 모듈이다. Gradle 설정, AndroidManifest, Java 코드, XML 리소스가 모두 이 안에 들어간다.

왜 분리했나: 루트는 프로젝트 관리용으로 두고, Android 런타임에 필요한 것은 app 안에 모아 빌드 단위를 명확하게 하기 위해서다.

app/src/main/java/com/example/myapplication/

화면 진입점과 Fragment

AuthGateActivity, LoginActivity, MainActivity, HomeFragment, GroupsFragment 같은 화면 코드가 있다.

왜 분리했나: 로그인, 팀 선택, 메인 탭, 그룹 화면처럼 사용자가 보는 흐름을 화면 단위로 빠르게 찾고 수정하기 위해서다.

app/src/main/java/.../data/

데이터 계약 계층

AuthRepository, UserRepository, GroupRepository, InningRepository 같은 인터페이스와 callback 타입이 있다.

왜 분리했나: 화면이 Firebase SDK를 직접 알지 않게 하려는 경계다. 화면은 "그룹을 만들어줘"라고 요청하고, 어디에 저장하는지는 구현체가 결정한다.

app/src/main/java/.../data/firebase/

Firebase 구현체

FirebaseGroupRepository, FirebaseInningRepository, FirebaseUserRepository가 실제 Auth, Firestore, Storage 호출을 한다.

왜 분리했나: Firebase 경로, batch write, upload 같은 외부 서비스 의존성을 한곳에 묶어 화면 코드가 복잡해지는 것을 막기 위해서다.

app/src/main/java/.../domain/

앱 규칙과 판단

GroupPermissionPolicy, GroupRoleResolver, VideoUploadPolicy, GroupFormValidator 등이 있다.

왜 분리했나: owner/admin/member 권한, 입력 검증, 영상 길이 제한 같은 규칙은 화면도 DB도 아니다. 별도 계층에 두면 같은 규칙을 여러 화면에서 재사용할 수 있다.

app/src/main/java/.../model/

데이터 모양

UserProfile, Group, GroupMember, Inning, InningRecord, Post 같은 값 객체가 있다.

왜 분리했나: Firestore 문서와 앱 내부 데이터의 모양을 맞춰 두면 저장, 조회, 화면 표시가 덜 헷갈린다.

app/src/main/java/.../util/

공통 도구

Ui는 카드/버튼/입력창 생성 helper이고, TeamCatalog는 구단 목록을 제공한다.

왜 분리했나: 반복되는 UI 생성과 팀 데이터를 화면마다 복붙하지 않기 위해서다. 다만 UI 크기와 radius도 여기서 많이 정해진다.

app/src/main/res/

Android 리소스

layout, menu, drawable, values, xml이 있다. XML 화면, 하단 탭, 색상, 문자열, FileProvider 경로를 관리한다.

왜 분리했나: Android는 코드와 리소스를 분리해 화면 구조, 색상, 문자열, 아이콘을 독립적으로 관리한다.

docs/

기획과 결정 기록

MVP 계획, Firebase 설정 기록, UI 비교 HTML, 오류 문구 정책 같은 프로젝트 지식이 들어간다.

왜 분리했나: 코드만 보면 왜 이렇게 만들었는지 알기 어렵다. 과거 결정과 현재 상태를 문서로 남겨 다음 수정 때 기준점을 제공한다.

outputs/

검증 산출물

실기기 캡처, XML dump, Firebase CLI cache 같은 작업 결과물이 있다.

왜 분리했나: 앱 코드와 검증 결과를 분리해, 화면 변경 전후와 Firebase 검증 증거를 나중에 다시 볼 수 있게 하기 위해서다.

주요 데이터 흐름

1. 로그인 후 앱 진입

  1. AuthGateActivity가 현재 로그인 상태를 확인한다.
  2. UserRepository.getCurrentUser()로 Firestore의 사용자 프로필을 읽는다.
  3. 선호팀이 없으면 TeamSelectActivity, 있으면 MainActivity로 이동한다.
  4. MainActivity는 프로필의 favoriteTeamId를 기준으로 홈 화면을 연다.

2. 하단 탭 화면 전환

  1. activity_main.xml의 BottomNavigationView가 탭 클릭을 받는다.
  2. MainActivity가 선택된 메뉴 id를 보고 Fragment를 만든다.
  3. fragment_container에 replace()로 홈, 응원방, 그룹, 응원가, 마이 화면을 갈아끼운다.
  4. 공통 헤더와 팀 칩은 Activity에 남고, 화면 내용만 Fragment가 담당한다.
그룹 목록 조회

GroupsFragment.loadGroups() → GroupRepository.getGroupsByTeam() → FirebaseGroupRepository → Firestore groups 조회 → ListCallback으로 돌아와 카드 렌더링.

그룹 생성

GroupsFragment.createGroup() → GroupFormValidator로 이름 검증 → FirebaseGroupRepository.createGroup() → Firestore batch로 groups/{groupId}와 groups/{groupId}/members/{uid} 동시 저장.

회차 기록 저장

GroupsFragment.saveMyRecord() → InningRecordFormValidator 검증 → FirebaseInningRepository → 미디어가 있으면 Storage 업로드 → Firestore records/{userId} 저장 → 상세 화면 다시 로드.

선호팀 저장

TeamSelectActivity.selectTeam() → UserRepository.updateFavoriteTeam() → Firestore users/{uid}.favoriteTeamId 업데이트 → MainActivity 진입.

파일별 빠른 찾기표

찾고 싶은 것 어디를 보면 되는가 데이터 흐름상 역할
앱 첫 진입과 로그인 분기 AuthGateActivity.java 로그인 여부와 선호팀 여부를 보고 다음 Activity를 고른다.
하단 탭과 공통 헤더 MainActivity.java, activity_main.xml 프로필과 선택 팀을 들고 Fragment를 교체한다.
홈 화면 문구와 팀 요약 HomeFragment.java, HomeFeedFormatter.java 선택 팀 id를 받아 순위, 최근 전적, 직전 경기 문구를 만든다.
그룹 목록, 생성, 상세, 회차 GroupsFragment.java 현재는 그룹 기능 대부분이 이 파일에 모여 있다. 향후 목록/생성/상세로 분리할 대상이다.
Firebase 그룹 저장/조회 FirebaseGroupRepository.java Firestore groups와 members 문서를 실제로 읽고 쓴다.
회차 기록과 미디어 업로드 FirebaseInningRepository.java Storage 업로드 후 Firestore record 문서를 저장한다.
권한 판단 GroupPermissionPolicy.java, GroupRoleResolver.java owner/admin/member가 무엇을 할 수 있는지 UI와 Repository 앞단에서 판단한다.
공통 카드/버튼 크기 Ui.java 프로그래밍 방식으로 만든 UI의 radius, padding, 버튼 높이, 카드 스타일을 결정한다.
Firebase 보안 강제 firestore.rules, storage.rules 앱을 우회한 요청도 서버 쪽에서 최종 차단한다.

현재 구조에서 알아둬야 할 점

이 구조의 장점은 Firebase 접근을 Repository에 모아 화면 코드가 어느 정도 분리되어 있다는 점이다. 반대로 GroupsFragment는 목록, 생성 폼, 상세, 초대, 회차, 업로드까지 너무 많은 책임을 들고 있다. 그래서 그룹 생성과 그룹 상세를 별도 Activity 또는 별도 Fragment로 분리하면 이후 수정이 훨씬 편해진다.
